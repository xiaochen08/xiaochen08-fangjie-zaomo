## 安装 Skill

这个仓库根目录就是可安装的 Skill，核心入口为 [SKILL.md](SKILL.md)。

在 Windows PowerShell 中执行：

~~~powershell
python -X utf8 "$env:USERPROFILE\.codex\skills\.system\skill-installer\scripts\install-skill-from-github.py" --repo xiaochen08/MC-FJZM --path . --name create-blockbench-minecraft-models
~~~

安装完成后重启 Codex，然后直接说：

~~~text
使用 $create-blockbench-minecraft-models 帮我制作一个 Minecraft Java 模型。
先询问需求并生成三套方案图，没有得到我的明确批准前不要正式建模。
~~~

Skill 本体包含：

| 目录 | 作用 |
|---|---|
| [SKILL.md](SKILL.md) | Codex 识别和执行的主入口 |
| [agents/openai.yaml](agents/openai.yaml) | 技能名称、简介和默认提示 |
| [references](references) | 模型、动画、粒子、音效、Mod 和交付规则 |
| [scripts](scripts) | 模型、音频、项目与交付验证脚本 |
| [scripts/tests](scripts/tests) | 自动化测试 |

## v1.1.0 重点更新：模型不能只做出来，还要能进游戏

以前用户选择“先做模型、以后再做 Mod”时，最大的风险是：外观看起来完成了，但实体类型、渲染方式、动画运行库、发射挂点和联动事件都没确定，最后可能无法接入游戏。

v1.1.0 增加了正式的运行时可落地门槛：

- 静态装饰、普通物品等低风险资产仍然可以先做模型；
- Boss、动态炮塔、复杂生物、投射物、粒子/音效联动和多人同步资产，默认推荐先创建最小 Mod 壳；
- 用户坚持先模型时，必须确认运行目标、风险和最高生产阶段；
- 关键运行参数没有确定时，只能制作概念图或灰盒，不能直接生产所谓成品；
- 正式生产前必须生成并验证 `runtime-contract.json`；
- 后期接入 Mod 时使用 `integration-map.json` 对齐骨骼、动画、事件、挂点、渲染器、服务端逻辑和客户端效果；
- 没有真实 Mod 工程、构建和游戏测试，就不能声称“已经可以进游戏”。

## 这个 Skill 到底强在哪？

说大白话：你负责说想法和做选择，Skill 负责把问题问清楚、先给你看方案、再去建模。

它不会听到“做个怪兽”就直接打开 Blockbench 乱做。正式建模前，它必须先确认下面这些事：

- ✅ 你要做的是生物、武器、机器、建筑，还是别的东西；
- ✅ 你是只做模型，还是先创建 Java Mod 再做模型；
- ✅ 模型长什么样、用什么材质、需要多高的纹理精度；
- ✅ 要不要待机、移动、攻击、受伤、死亡、损毁等动画；
- ✅ 要不要粒子、发光、能量弹和音效；
- ✅ 周边还要不要武器、掉落物、召唤物、底座或配套建筑；
- ✅ 文件放在哪个盘、属于哪个模型，避免做多以后互相搞混。

需求问完以后，它会先生成 **A、B、C 三套能在 Blockbench 里做出来的方案图**。你选中并明确说“按 B 方案开始建模”以后，它才会制作正式模型。

这就是它最重要的能力：**先让你看懂、选对，再动手。** 🚀

## 没有 Mod，可以先做模型吗？

可以，但不是所有模型都适合盲目先做。

| 风险 | 例子 | 建议 |
|---|---|---|
| 低 | 静态装饰、简单物品、家具 | 可以先做模型 |
| 中 | 简单动画生物、可穿戴物、基础载具 | 推荐先确定 Mod 技术路线 |
| 高 | Boss、动态炮塔、方块实体、能量弹、攻击判定、粒子音效、多玩家同步 | 默认先创建最小 Mod 壳 |

如果高风险模型暂时没有 Mod，Skill 会先生成运行时合同，把模型身份、骨骼、动画、事件和挂点固定下来；如果渲染方式或动画运行库仍然不明确，就停在概念图或灰盒，不会制造一堆看似完整却无法落地的文件。

## 三个真实例子

### 例子一：机械三头狼 Boss 🐺

你可以这样说：

> 做一个黑铁材质的机械三头狼 Boss，眼睛发红光。要待机、扑咬、左右头攻击、受伤、死亡动画，死亡后保留损坏模型。

Skill 会继续问清楚体型、纹理精度、三个头怎么分工、红光是否需要粒子、死亡后是倒地还是散架。确认后先给三套 Blockbench 风格方案图，不会直接开做。

### 例子二：水晶能量塔 ⚡

你可以这样说：

> 做一个会发射能量弹的水晶塔。中间能量从下往上输送，两块护盾板围着水晶转，停火 5 秒后再慢慢归位。

Skill 会重点检查旋转中心、护盾板与水晶的安全距离、攻击衔接和冷却时间，避免出现护盾板自己打转、方向相反、重叠或穿模。

### 例子三：会升级的魔法巨剑 🗡️

你可以这样说：

> 做一把三阶段升级的魔法巨剑。普通状态没有特效，蓄力时符文亮起，满级时出现蓝色火焰和环形粒子。

Skill 会把普通、蓄力、满级和损坏状态分开设计，并确认它是手持武器、展示物，还是两种用途都要。

## 整个过程很简单

~~~text
你说想法
   ↓
Skill 问清楚用途、版本、保存位置和配套内容
   ↓
确认纹理、动画、粒子、音效和损毁方式
   ↓
生成 A / B / C 三套方案图
   ↓
你选一套并明确批准
   ↓
Blockbench 建模、贴图、骨骼和动画
   ↓
检查穿模、转轴、文件归属和运行效果
   ↓
检查运行时合同和允许生产到哪一步
   ↓
接入真实 Java Mod、构建并进游戏验证
~~~

## 纹理选多大？

| 精度 | 适合什么 | 会发生什么 |
|---|---|---|
| 64×64 | 小物品、简单方块模型 | 文件小，但细小纹路不明显 |
| 128×128 | 大多数普通模型 | 清楚、够用，成本比较平衡 |
| 256×256 | Boss、机械体、重点道具 | 能画更多磨损、接缝和发光细节 |
| 512×512 | 大型展示模型、近距离特写 | 最细，但制作和修改更费时间 |

高像素不等于自动变好。模型结构很简单时，512×512 只会让文件更大；如果选择高精度，Skill 也必须按高精度标准把材质、阴影、接缝和高光做完整。

## 一个模型一个文件夹 📁

每个模型都有自己的独立工作区：

~~~text
你的项目目录/
└─ crystal_tower/
   ├─ model/
   ├─ textures/
   ├─ animations/
   ├─ audio/
   ├─ particles/
   ├─ previews/
   ├─ mod/
   └─ evidence/
~~~

这样即使同时制作十个模型，水晶塔的动画和音效也不会误绑到三头狼身上。

## 你只要记住五句话

1. **没问清楚，就不建模。**
