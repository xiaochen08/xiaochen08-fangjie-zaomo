# 从一句想法到游戏模型：完整流程 🧭

这份文档不讲复杂名词，只说每一步要做什么、为什么要做。

## v5.2 的四个 Skill 怎么配合

`$fjzm` 是项目负责人，`$fjzm-model` 是模型几何专修，`$fjzm-texture` 是纹理专修，`$fjzm-animation` 是动画与战斗编排专修。不是四个 Skill 同时随便改同一个模型，而是由主控按 ContractFlow v1 顺序交接：

~~~text
$fjzm 锁定模型与用户批准
  → 生成并验证 model-handoff.json
  → $fjzm-model 制作灰盒、几何、基础骨架与八视图证据
  → 返回 model-result.json
  → 生成 texture-handoff.json
  → $fjzm-texture 制作或返修贴图
  → 返回 texture-result.json
  → 生成并验证 animation-handoff.json
  → $fjzm-animation 独占一个动画版本进行制作或返修
  → 返回 animation-result.json
  → $fjzm 核对结果并重新绑定粒子、音效、碰撞、投射物与 Mod
~~~

每次交接都要核对项目 ID、模型 ID、版本、模型哈希和骨骼签名。主控是唯一调度者，三个工坊不能越权互相调用或改写对方负责的文件。动画工坊永远写新版本，不覆盖源文件。如果问题其实来自几何、UV、纹理或骨骼拓扑，它会停止并退回主流程，而不是擅自拆模型。

## 第 1 步：先锁定 Minecraft 版本，再决定做模型还是先建 Mod

Skill 的第一个问题固定是 Minecraft Java 版本，例如 `1.20.1` 或 `1.21.1`。在版本没有锁定和记录官方证据前，不询问资产外观，也不创建源码。

版本确认后，Skill 才会问：

> 你已经有 Java Mod 项目了吗？
>
> 如果没有，你想先搭 Mod，再做模型；还是先把模型做好，以后再接 Mod？

两条路都可以，但不是所有模型风险都一样：

- **先建 Mod**：模型尺寸、命名、事件和导出格式能早点定下来，后面接入更顺。
- **先做模型**：适合静态装饰和早期创意验证；复杂模型必须先通过运行时风险门槛。

Skill 不能替你偷偷选，必须等你明确回答。

### 没有 Mod 时的风险分级

| 等级 | 常见内容 | 允许做到哪里 |
|---|---|---|
| 低 | 静态方块、简单物品、家具 | 可制作运行时中立模型源 |
| 中 | 简单动画实体、可穿戴状态、基础载具 | 推荐先建 Mod；拒绝后需确认风险 |
| 高 | Boss、炮塔、方块实体、投射物、粒子音效、攻击伤害、多人同步 | 默认先建最小 Mod；目标不明时只能做到概念或灰盒 |

选择模型优先后，Skill 会生成 `runtime-contract.json`，记录游戏身份、版本状态、渲染方式、动画运行时、骨骼签名、动画 ID、事件 ID、发射挂点和最高生产阶段。这个合同验证失败，就不能继续细化。

后来有了真实 Mod 工程，Skill 会先检查工程，再生成 `integration-map.json`，把每个模型接口对应到 Java 注册、渲染器、动画控制、服务端攻击、客户端粒子音效、网络同步和存档逻辑。只有构建并进游戏测试后，才能称为正式落地。

### 生图前的光影门槛

Skill 还会在出 A/B/C 方案图之前问：

- 关闭光影时，模型是否必须保持正常可读（默认必须）；
- 使用 Iris、OptiFine，还是其他加载器，准确版本是什么；
- 是否指定某个光影包、版本和画质预设；
- 是否需要 PBR，采用哪种材质标准和通道规则；
- 发光只是自己变亮，还是还要真的照亮附近方块；
- 材质是实心、镂空还是半透明，使用哪个渲染层；
- 目标电脑性能、观察距离和可接受的贴图/透明层成本。

不知道具体光影包时，Skill 会按无光影保底做中性材质，不会假装支持全部光影。方案图不会把固定方向的强高光、阴影或泛光画死；正式贴图前必须生成 `shader-contract.json`。真正的光影兼容必须用准确版本进游戏测试。

## 第 2 步：确认做什么

先选大类，再说细节：

- 生物、动物、怪物、Boss；
- 武器、盔甲、工具、道具；
- 机器、炮台、生电装置；
- 家具、装饰、建筑；
- 载具、机甲、飞船；
- 植物、地形、环境物件。

然后再问：它在游戏里干什么、玩家从多远看到它、是不是经常拿在手里、要不要近距离观察。

## 第 3 步：顺手把周边内容想清楚

一个 Boss 可能还需要：

- 蛋、召唤物或生成方式；
- 专属武器和掉落物；
- 攻击弹、法阵、护盾；
- 完整版、受损版、死亡残骸；
- Boss 房间或展示底座。

这一步会在生图前确认。选了的配套内容会一起出现在方案图里，没选的不会偷偷加。

## 第 4 步：给模型一个身份证

每个模型都要有唯一名称和独立文件夹，例如：

~~~text
asset_id: crystal_tower
中文名: 水晶能量塔
工程根目录: D:/FJZM-Projects/energy_defense/
模型目录: D:/FJZM-Projects/energy_defense/assets/models/crystal_tower__v1/
~~~

后面的贴图、动画、粒子、音效和 Mod 文件都必须写同一个 asset_id。这样多模型同时制作时不会混。

## 第 5 步：说清楚外观

Skill 内部会保存完整问题队列，但每次对话只问一个最关键的问题，不会一次甩出一整张专业表格：

1. 用途、版本、体型和比例；
2. 轮廓、材质、颜色和最重要的辨识点；
3. 动画、粒子、音效、损毁方式和参考图。

光影、PBR、发光和透明层会并入这三组问题，不会等建模完成后再补问。

用户回答不完整时，只补问会影响方案的内容。

## 第 6 步：选纹理精度

| 选项 | 大白话解释 |
|---|---|
| 64×64 | 能看清大色块，细小划痕会糊在一起 |
| 128×128 | 普通模型最常用，效果和工作量比较平衡 |
| 256×256 | 适合 Boss、机械体、重点道具 |
| 512×512 | 适合大型展示和近距离特写，制作成本最高 |

Skill 必须告诉用户：低精度会丢什么，高精度会多花什么，以及这个模型到底需不需要高精度。

## 第 7 步：把动画说成人话

动画不能只写一个“攻击”。至少要说清：

- 什么时候开始；
- 哪个部位先动；
- 动作持续多久；
- 命中发生在哪一帧；
- 粒子和音效什么时候触发；
- 攻击结束后立刻归位，还是等几秒；
- 中间怎么衔接；
- 有没有可能穿模。

### 水晶塔例子

~~~text
启动：两块护盾板立刻展开。
攻击：护盾板围绕水晶中心公转，不是自己原地转。
能量：方块能量条从底部一格一格送到水晶。
发射：水晶偏转一次，射出一枚能量弹。
预冷：最后一次攻击后继续环绕 5 秒。
归位：速度逐渐变慢，再回到左右两侧。
检查：整段动画中护盾板不能碰到水晶。
~~~

## 第 8 步：确定粒子和音效

粒子要写清颜色、形状、出现位置、方向、数量和持续时间。

音频可以直接拖进对话框。用户可以用中文文件名或编号，例如：

~~~text
01_启动.wav
02_能量上升.wav
03_发射.wav
04_冷却.wav
~~~

Skill 会保留原文件，同时复制出安全英文名：

~~~text
01_start.wav
02_energy_rise.wav
03_fire.wav
04_cooldown.wav
~~~

正式绑定前必须给用户看一张“事件—音效”对应表。顺序不对、模型身份不对，都不能继续。

## 第 9 步：按轮次生图并永久留档

多资产项目先生成全资产总览，再生成主模型 A、B、C 三套不同方向。主题选定后，继续按单资产细化、完整视图、动作关键帧、GUI 主题和单屏状态逐轮确认。

这些图必须看起来像 Blockbench 真能做出的结果：

- 以立方体和可实现的斜面为主；
- 不假装模型有软件做不到的曲面；
- 不用大量电影级烟雾遮住结构；
- 能看清关节、骨骼、转轴和部件间距；
- 明确展示纹理像素感和实际模型比例。

最终模型参考必须包含正、背、左、右、顶、底和三分之四视图；有动画时还要生成独立动作关键帧图。模型、动作和 GUI 不挤在同一张图里。

每轮图片、提示词、负面提示词、审查、修改意见、批准证据和哈希都保存在 `design/image-rounds/`。下一次对话读取 `image-production-index.json` 继续未完成轮次，旧图不覆盖。

## 第 10 步：等用户明确批准

以下说法才算批准：

> 我选择 B，按 B 的造型开始正式建模。

“还可以”“先看看”“差不多”都不算批准。没有明确批准，不打开 Blockbench，不生成正式模型。

## 第 11 步：正式制作

批准后才进入：

1. 灰模和尺寸；
2. 骨骼、分组和转轴；
3. 动画测试；
4. 纹理和发光区域；
5. 粒子、音效事件；
6. 损毁版本；
7. 导出和 Mod 接入。

每一阶段都必须沿用用户批准的方案，不能在制作中偷偷改造型。

## 第 12 步：检查后交付 ✅

交付前至少检查：

- 文件能不能打开；
- 贴图有没有丢；
- 动画名称和事件有没有写错；
- 转轴是不是放在真正的中心；
- 部件有没有重叠或穿模；
- 粒子和音效是不是绑到正确模型；
- Mod 能不能构建和启动；
- 每个交付结果有没有截图、日志或报告作为证据。

**工具说“成功”不等于真的完成。看得到、打得开、跑得动，才算完成。Enjoy! 🎮**
