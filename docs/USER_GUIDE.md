# 新手使用教程：照着说就行 📖

你不需要会建模，也不需要记专业词。按下面的例子说，Skill 会继续带你走。

## 怎么启动？

在 Codex 对话里直接说：

> 使用方界造模，帮我做一个 Minecraft Java 版模型。

如果你的环境支持技能名，也可以明确写：

~~~text
$fjzm
~~~

正式中文名是“方界造模”，品牌缩写是 `FJZM`。由于 Skill 调用名必须使用小写字符，所以在对话中调用时写 `$fjzm`。

v3.0.0 会同时安装两个入口：

- 平时从零做模型、接 Mod、统一处理贴图/光影/粒子/音效时，用 `$fjzm`；
- 模型已经确认，只想诊断、重做或细调骨骼和动画时，用 `$fjzm-animation`。

你不需要自己在两个 Skill 之间搬文件。主流程会生成 `animation-handoff.json`，动画工坊按合同修改新版本，再用 `animation-result.json` 交回主流程。只有结果通过后，粒子、音效、碰撞、能量弹和 Mod 才能跟着新动画重新绑定。

如果你单独调用动画工坊，可以直接说：

~~~text
使用 $fjzm-animation 检查这个 .bbmodel 的攻击动画。
先告诉我问题是时间、关键姿势、插值、旋转中心、穿模距离，还是模型结构本身；
给出返修方案让我确认，不要覆盖原文件，也不要修改没有批准的模型结构。
~~~

## 第一次怎么说最省事？

复制下面这段，把括号内容换成自己的：

> 我要做一个【模型类别】，用于 Minecraft Java【版本】。
>
> 我【已经有 / 还没有】Mod 项目。
>
> 模型想做成【外观和材质】，需要【动画】。
>
> 文件想放到【盘符或文件夹】。
>
> 先问清楚需求，再给我三套方案图，我确认后才能正式建模。

## Skill 会先问什么？

通常会问这几类：

1. **项目路线**：先建 Mod，还是先做模型；
2. **模型类型**：Boss、武器、机器、建筑等；
3. **保存位置**：放哪个盘、文件夹叫什么；
4. **外观**：尺寸、颜色、材质、重点部件；
5. **配套内容**：武器、掉落物、底座、破损版；
6. **动画**：待机、攻击、受伤、死亡、损毁；
7. **效果**：粒子、发光、能量弹、音效；
8. **精度**：64、128、256 还是 512 像素。
9. **光影**：无光影保底、Iris/OptiFine、指定光影包版本、PBR、发光和透明层。

如果你没有 Mod，Skill 还会判断运行时风险：

- 静态装饰和简单物品通常可以先做模型；
- Boss、动态炮塔、复杂生物、投射物、攻击判定、粒子音效或多人同步内容，默认建议先搭最小 Mod；
- 你坚持先模型时，需要明确接受“平台导出延期、后续可能需要适配、目前不能称为游戏可用”；
- 渲染方式或动画运行时没确定时，Skill 只能做到概念图或灰盒。

## 光影要怎么说？

不懂专业词也没关系，可以直接说：

> 我希望不开光影也正常；主要使用 Iris。先按普通 Minecraft 像素材质做，不确定 PBR。水晶需要自己发亮，但要不要照亮周围方块请你说明两种方案。半透明材质如果容易穿帮，就优先改成镂空贴图。

如果你有明确目标，再补充光影包准确名称、版本和预设。Skill 会把这些写入 `shader-contract.json`。它不会承诺“所有光影都一样好看”，只会验证无光影保底和你指定的目标。

发光有两件不同的事：

- **视觉发亮**：用发光遮罩或目标材质通道让水晶/眼睛在暗处保持明亮；
- **照亮环境**：附近地面和方块真的变亮，需要 Mod 的光照或动态光源逻辑，单靠模型贴图做不到。

正式验收会检查白天、暗处、侧光、Bloom、透明重叠，以及指定光影包的实际游戏画面。概念图和 Blockbench 截图不能代替这一步。

## 例子一：机械三头狼 Boss 🐺

### 你可以这样说

> 我要做一个机械三头狼 Boss，黑铁材质，眼睛红色发光。体型大约是玩家的两倍。需要待机、走路、三种攻击、受伤、死亡和损毁动画。死亡后不要直接消失，留下冒烟的机械残骸。纹理先按 256×256 评估。

### Skill 应该继续确认

- 三个头是一起攻击还是轮流攻击；
- 中间头是不是主要攻击头；
- 红眼是贴图发光还是还要红色粒子；
- 死亡残骸是一整块倒下，还是零件散开；
- 是否需要 Boss 蛋、掉落核心和专属武器。

### 生图后怎么批准

> 我选择第二套。保留第二套的头部和装甲，后腿再粗一点。修改图确认后，再按第二套正式建模。

## 例子二：水晶能量塔 ⚡

### 你可以这样说

> 做一个黑铁和青蓝水晶组成的能量塔。塔身内部有方块能量条，从底部一直向上输送。攻击时两块护盾板围着水晶公转，水晶偏转后发射能量弹。停止攻击后护盾继续转 5 秒，再慢慢归位。

### 必须说清的细节

- 护盾板围绕水晶中心转，不是护盾板自己原地转；
- 两块护盾板保持相同朝向和固定间隔；
- 旋转半径要足够大，任何一帧都不能碰到水晶；
- 能量条、水晶发光、发射音效和能量弹必须按正确顺序触发；
- 攻击结束不是马上归位，而是先等 5 秒。

### 推荐事件顺序

| 时间 | 发生什么 |
|---|---|
| 0.0 秒 | 护盾展开，播放启动声 |
| 0.3 秒 | 能量条开始向上移动 |
| 0.8 秒 | 水晶变亮，护盾开始公转 |
| 1.2 秒 | 水晶偏转，发射能量弹 |
| 停火后 0–5 秒 | 护盾保持环绕，水晶逐渐变暗 |
| 停火后 5–6.5 秒 | 护盾减速并回到左右两侧 |

## 例子三：三阶段魔法巨剑 🗡️

### 你可以这样说

> 做一把玩家双手使用的魔法巨剑。基础状态是暗银材质；蓄力时剑身符文从下往上亮起；满蓄力后出现青蓝火焰粒子；耐久归零时显示裂纹和断刃模型。

Skill 会确认手持比例、第一人称和第三人称显示、符文点亮顺序、满蓄力声音、断裂位置，以及是否需要展示架。

## 纹理怎么选？

- **64×64**：简单、快，适合小物件；
- **128×128**：大多数模型都够用；
- **256×256**：Boss、机械体和重点武器更合适；
- **512×512**：只有大型近景展示真的需要。

如果你说“我要最高精度”，Skill 不能只把画布放大。它还必须把材质变化、划痕、接缝、阴影和高光做到相应水平。

## 音频怎么交？

直接把 wav 或 ogg 文件拖进对话框。中文名和编号都可以：

~~~text
01_启动
02_蓄力
03_发射
04_命中
05_冷却
~~~

Skill 会整理英文安全文件名，再让你确认每个声音绑在哪个事件上。确认前不会自动绑定。

## 修改意见怎么说最清楚？

不要只说“有点怪”。按这个顺序说：

~~~text
对象：左侧护盾板
问题：旋转时碰到水晶
正确效果：围绕水晶中心公转
修改数值：半径增加 3 个 Blockbench 单位
检查范围：完整攻击和 5 秒冷却动画
~~~

## 最后你会拿到什么？

根据项目内容，通常会包括：

- Blockbench 模型源文件；
- 纹理图；
- 动画和事件表；
- 粒子设计；
- 整理后的音频和映射表；
- 完整、受损、损毁版本；
- Mod 接入文件；
- 光影兼容合同、发光/PBR 材质图和游戏内测试证据；
- 预览图、测试日志和交付报告。

## 四条防翻车规则 ✅

1. 三套图没看懂，就继续改图，不要急着建模。
2. 只有明确说“按某套开始建模”才算批准。
3. 多模型制作时，先检查当前 asset_id。
4. 交付时一定要看截图、动画预览和验证报告。

照着这份教程说，就算第一次用 Blockbench，也能把流程走明白。Enjoy creating! ✨
